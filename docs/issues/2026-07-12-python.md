# Python bindings review — issues and improvements

Date: 2026-07-12. Scope: `crates/fiml-python` (PyO3 bindings), reviewed against
`docs/specs/python.md`. Status values: `open`, `in-progress`, `done`.

## 1. Broken / stale (blockers)

### 1.1 Bindings do not compile after the core rename — `done`

`crates/fiml-python/src/lib.rs:9` still imports `DynIndicatorEngine` and
`EngineSpec`; the core crate now exports `FeatureExtractor` and `FeatureSet`
(spec §12.5/§12.6). `from_spec` is now `from_feature_set`.
`cargo check -p fiml-python` fails with E0432.

### 1.2 Stale `"spec"` JSON key in examples and README — `done`

The core `FeatureDef` field was renamed `spec` → `indicator`, which changed the
per-feature JSON key. Still using the old key (would fail to deserialize at
runtime even after 1.1 is fixed):

- `crates/fiml-python/examples/quickstart.py` (`SPEC` dict)
- `crates/fiml-python/README.md` (parity-contract JSON example)
- `crates/fiml/examples/spec_engine.rs` (`SPEC_JSON` literal)

### 1.3 Naming not yet aligned with spec §12.5/§12.6 — `done`

Python class is still `Engine` with `from_spec_json`. Target: `FeatureExtractor`
with `FeatureExtractor.from_json` / construction from a `FeatureSet`, and
matching updates in `python/fiml/__init__.py`, quickstart, README, docstrings.

## 2. Correctness

### 2.1 Warmup rows read as 0.0 — `done`

Feature cells initialize to zero and are only written once an indicator has a
value, so `transform` rows emitted before warmup contain `0.0` —
indistinguishable from a legitimate zero (e.g. OBV). A model trains on fake
zeros silently. Spec recommendation: NaN warmup. Options, cheapest first:

- initialize output cells to NaN in the bindings' snapshot (mask never-written
  cells),
- have the core initialize the feature vector to NaN,
- return a warmup-row count/mask alongside the matrix.

Must be resolved before anyone trains on `transform`/`compute_features` output.

### 2.2 Partial state mutation when `transform` fails mid-stream — `done`

If row `i` fails (missing payload column for its kind, unknown symbol handle),
rows `0..i` have already advanced the extractor's state, leaving it
half-stepped with no way to tell. Fix: pre-validate before dispatching — one
cheap pass over `kind` checking each kind present has its required columns and
each symbol handle is in range — giving all-or-nothing semantics.

## 3. Missing API (spec gaps)

### 3.1 No `FeatureSet` builder — users hand-write serde JSON — `done`

`from_spec_json` forces Python users to write Rust-enum-tagged JSON, including
`Duration` as `{"secs": ..., "nanos": ...}`. Spec §3b: a `FeatureSet` pyclass
with a fluent builder (`.sma()`, `.ema()`, `.sma_timed()`, `.obv_timed()`,
`.trade_count_timed()`, `.day_of_week()`, `.time_since_session_open()`) plus
`to_json()` / `from_json()`. JSON stays the parity artifact saved next to the
model; humans stop writing it. Highest-value improvement.

Note: core deviates from the spec's `tz: str` — the Rust variant is
`TimeSinceSessionOpen { utc_offset_millis: i64 }` to avoid an IANA tz-database
dependency (AGENTS.md: minimal third-party dependencies). The Python builder
should map `tz="UTC"` → `0`.

### 3.2 No high-level `compute_features(df, ...)` — `done`

Spec §6. Today users must manually build `kind`/`symbol`/`timestamp` arrays and
know the kind codes. Needed: a thin layer (Python-side is fine) taking a
dataframe, `source="bars"|"trades"`, column-name kwargs; converts
`datetime64[ns]` → epoch ms, interns the symbol column, returns one feature row
per input row. Low-level `transform` stays as the escape hatch for mixed event
streams.

## 4. Performance / ergonomics

### 4.1 `values()` allocates per call — `done`

Returns `Vec<f64>` → Python list; in a streaming loop that is an allocation per
event. Return a numpy array directly, or add an `update` variant writing into a
caller-provided buffer (fits AGENTS.md "minimal memory allocation").

### 4.2 GIL held for the whole `transform` loop — `open`, low priority

For large frames, `py.allow_threads` around the dispatch loop would let other
Python threads run. Needs restructuring (numpy borrows are GIL-tied). Not
urgent for the single-threaded ML use case.

### 4.3 No `reset()` — `open`, needs core support

Re-running a backtest requires re-parsing JSON to get fresh state. A core-side
state reset exposed in the bindings would avoid that. Core change — confirm
before implementing.

### 4.4 Cosmetics — `open`

- `symbol()` returns `usize` while the `symbol` column dtype is `i64`.
- pyo3/numpy pinned at 0.23 — a few releases behind; bump when convenient.

## 5. Packaging / tooling

### 5.1 No `.pyi` stubs / `py.typed` marker — `open`

ML users live in notebooks and IDEs; typed stubs for
`FeatureExtractor`/`FeatureSet` are cheap and high leverage.

### 5.2 No pytest suite — `open`

`examples/quickstart.py` doubles as the only test. A real `tests/` (parity,
error cases, warmup) would have caught the stale-JSON breakage in 1.2.

### 5.3 No CI wheel build — `open`

No maturin build in CI; breakage like 1.1 only surfaces locally.

## Considered and rejected

Per-kind batch methods (`transform_prices(...)`, `transform_trades(...)`) would
remove the per-row `Option` checks and the `kind` column, but they destroy the
single interleaved event stream in true time order that the parity contract
depends on. Keep the unified `(kind, symbol, timestamp, payload...)` layout as
the low-level API under `compute_features`.
